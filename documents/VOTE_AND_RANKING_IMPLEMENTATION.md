# Ice Music Player - 点赞、点踩与排序功能实现技术总结

## 1. 核心设计理念

本项目采用了 **Redis (缓存/实时数据)** + **MySQL (持久化/全量数据)** 的 [Cache-Aside / Write-Through 混合] 策略，以保证高并发下的点赞性能，同时确保数据的最终一致性。

### 为什么这样做？
- **Redis Set**：利用 Redis 的 `Set` 数据结构存储 `likes:{songId}` 和 `dislikes:{songId}`，成员为 `userId`。
  - **优势**：
    - **天然去重**：不需要额外逻辑判断用户是否重复点赞，`SADD` 自动处理。
    - **性能极高**：点赞/取消点赞仅需 O(1) 复杂度。
    - **实时计数**：`SCARD` 可以瞬间获取当前绝对实时的票数。
- **MySQL**：作为数据的 Source of Truth，防止 Redis 宕机导致数据丢失。

## 2. 数据流转架构

### 2.1 用户操作流程 (VoteService)

当用户点击“点赞”或“点踩”时：

1.  **校验**：检查歌曲是否存在。
2.  **更新 MySQL (SongVote表)**：
    - 记录用户与歌曲的点赞关系表（`user_id`, `song_id`, `vote_type`）。
    - 这一步保证了用户个人的操作记录被持久化，后续用户刷新页面能看到自己“已点赞”的状态。
3.  **更新 Redis**：
    - 将 `userId` 加入（`SADD`）或移除（`SREM`）对应的 Redis Set。
    - 比如点赞：`likes:{songId}` 添加用户，同时从 `dislikes:{songId}` 移除用户（互斥逻辑）。
4.  **返回最新票数**：
    - 直接返回 Redis 中的 `SCARD` 计数，保证前端拿到的数字是最新的。

### 2.2 列表查询与排序流程 (SongService)

这是本项目的一个亮点，解决了“缓存一致性”问题。

1.  **前端请求**：用户切换歌单或打开页面，请求 `/api/public/songs/get`。
2.  **后端处理**：
    - **第一层：Redis 缓存 (List<Song>)**
        - 首先尝试从 Redis 获取缓存的歌单列表（含 OSS 签名 URL，有效期 6 小时）。
        - 缓存命中后不会直接信任其中的 `likeCount` / `dislikeCount`，而是先进行一次实时修正。
        - **实时修正 (Overlay)**：遍历列表中的每一首歌，再次查询 Redis `likes:{id}` 和 `dislikes:{id}` 的 `SCARD` 大小，覆盖到对象中。
        - 这样做的原因是：缓存里的 `Song` 对象主要承担歌曲基础信息和签名链接缓存职责，而票数以 Redis 中的实时集合计数为准。
    - **第二层：数据库兜底**
        - 如果缓存未命中，从 MySQL 查询列表，生成签名 URL，并写入缓存。
        - 在返回给前端之前，同样会进行一次**实时修正**，确保列表里的点赞数、点踩数与 Redis 中的最新值一致。
3.  **前端渲染 (Vue)**：
    - 前端拿到数据后，根据 `likeCount` 字段进行排序展示。

### 2.3 数据持久化同步 (VoteCountSyncTask)

Redis 虽然好用，但数据都在内存中。为了让 MySQL 的 `songs` 主表中的 `like_count` 和 `dislike_count` 字段也能反映真实数据（用于数据分析或 Redis 挂掉后的恢复）：

- **定时任务**：按固定周期执行 (`@Scheduled`)，并在应用启动时补做一次同步。
- **全量同步**：
    1. 遍历所有歌曲。
    2. 通过 Redis Pipeline 批量读取所有歌曲对应的 `likes:{songId}` 和 `dislikes:{songId}` 集合大小，减少大量小请求的往返开销。
    3. 将结果按顺序映射回歌曲列表，计算出每首歌当前应有的点赞数和点踩数。
    4. 只收集真正发生变化的歌曲，最后批量 `saveAll` 回写数据库。
- **启动同步**：项目启动时 (`@PostConstruct`) 会额外执行一次，避免 Redis 冷启动或缓存未预热时数据库中的票数字段长期滞后。

## 3. 前端交互实现

为了让用户体验如丝般顺滑，前端也做了精细化处理：

1.  **状态管理**：
    - `VoteControls.vue` 组件负责单个歌曲的点赞 UI。
    - 内部维护 `likes`, `dislikes`, `userVote` 三个响应式变量。
    - 点赞、点踩、取消投票后，组件会立即用接口返回值更新本地状态，保证当前歌曲区域的数字即时反馈。
2.  **实时通信 (EventBus)**：
    - 这是一个关键点。`VoteControls` 是 `Player` 的孙子组件，而 `Playlist` 是 `Player` 的子组件。
    - 当用户在播放器控制栏点赞时，`VoteControls` 会触发全局事件 `song-vote-updated`。
    - `Player.vue` 监听此事件，收到后直接修改内存中 `playlist` 数组对应歌曲的 `likeCount` / `dislikeCount` 属性。
    - **效果**：用户一点赞，无需刷新页面，左侧歌单列表里的点赞数立马跳动，且如果当前按点赞排序，歌单顺序也会瞬间更新。

---

## 4. 总结

| 模块 | 技术方案 | 作用 |
| :--- | :--- | :--- |
| **存储** | Redis Set | 实时、去重、高性能计数 |
| **持久化** | MySQL + 定时任务 | 数据最终一致性，防止丢失，并通过差量写回减少无意义更新 |
| **查询** | Cache Overlay (缓存覆盖) | 解决列表缓存与实时计数不一致的问题，保证返回列表中的票数始终接近实时值 |
| **交互** | EventBus (事件总线) | 组件间状态同步，实现UI即时响应 |

这套方案在保证了高性能（主要读写走 Redis）的前提下，完美解决了数据一致性和持久化的问题，是一个非常适合中小型高频互动应用的架构。
